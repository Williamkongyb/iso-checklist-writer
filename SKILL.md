# ISO Checklist Writer — 合规审查检查表自动生成 Skill

## 🎯 两种使用方案

### 方案A（推荐，常用）：完全离线运行
- **Token消耗**：0个 WorkBuddy token（完全不通过对话）
- **成本**：仅消耗 DeepSeek token（~15万/次，约¥0.3-0.5）
- **使用方法**：
  1. 打开文件夹 `C:\Users\Confu\.workbuddy\skills\iso-checklist-writer\`
  2. 双击 `启动检查表生成工具.bat`
  3. 按提示输入4个参数（客户记录目录、标准类型、审核日期、企业名称）

### 方案B（备选）：最小Token消耗模式

- **适用场景**：
  - 方案A不可用时（如批处理脚本损坏）
  - 需要通过对话触发时（如远程操作）
  - **调优时使用**：当方案A遇到问题，用方案B调优，然后发布A1, A2, A3等改进版本
- **调优流程**：

### Step 1: 一键生成检查表（方案B）

```bash
"C:\Users\Confu\.workbuddy\binaries\python\versions\3.13.12\python.exe" -u "C:\Users\Confu\.workbuddy\skills\iso-checklist-writer\run_all.py" --records-dir "D:/合规知识库-compliance-kb/records-客户记录/XXX" --standard ISO20000 --audit-date 2026-02-01 --company-name "企业名称" --quiet
```

**参数说明：**
- `--records-dir`: 客户记录目录路径（必需）

- `--audit-date`: 审核日期 YYYY-MM-DD（必需）
- `--company-name`: 企业名称（必需）
- `--template`: 模板文件路径（可选，默认自动选择）
- `--skip-read`: 跳过Step1，使用已有records.txt（可选）
- `--records-file`: 指定已有的记录文本文件路径（可选）
- `--quiet`: 安静模式，只输出最终结果，详细日志写入文件（可选，方案B推荐）

**输出文件：**
1. **检查表.docx**（格式克隆，输出到 `Skill目录\output\`）
2. **问题清单.txt**（DeepSeek生成，如果发现问题）

### Step 2: 交付文件（方案B）

## 🐛 已知问题与修复历史

### 问题1：Step1读取0个文件

- **现象**：`Step1: 0 files, 0 chars`
- **原因**：`priority_dirs` 硬编码了山东翰林的目录结构，其他客户目录结构不同
- **修复**：改为递归读取所有文件（`root.rglob("*")`）
- **状态**：✅ 已修复（2026-06-14）

### 问题2：输出目录权限被拒

- **现象**：`Permission denied: D:\合规知识库-compliance-kb\output-输出结果\...`
- **原因**：沙箱阻止Python写入D盘
- **修复**：`OUTPUT_DIR` 改为脚本本地 `output` 子目录
- **状态**：✅ 已修复（2026-06-14）
- **遗留**：需要用户手动复制到D盘，或授权沙箱写入

### 问题3：Windows GBK编码崩溃

- **现象**：`UnicodeEncodeError: 'gbk' codec can't encode...`
- **原因**：Python on Windows默认stdout编码为GBK，无法输出中文字符
- **修复**：添加 `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8', errors='replace')`
- **状态**：✅ 已修复（2026-06-13）

### 问题4：Python stdout缓冲无输出

- **现象**：脚本运行7分钟，无任何输出
- **原因**：Python默认缓冲模式，`print()` 不立即刷新
- **修复**：使用 `python -u` 参数运行（unbuffered mode）
- **状态**：✅ 已修复（2026-06-13）

### 问题5：`log()` 函数无限递归

- **现象**：运行脚本后卡死，无输出
- **原因**：批量替换 `print()` 为 `log()` 时，`log()` 函数内部调用被误替换为 `log()`，导致无限递归
- **修复**：手动修复 `log()` 和 `final_output()` 函数，将内部的 `log(msg, "INFO")` 改回 `print(msg, flush=True)`
- **状态**：✅ 已修复（2026-06-14）

## 📋 检查表输出规范（Rule 1-14）

> 以下内容来自项目级MEMORY.md，必须严格遵守！

### Rule 1：输出格式必须为Word文档

- 所有报告/检查表必须输出为 `.docx` 格式

### Rule 2：检查表必须严格按模板格式输出（格式克隆机器人）

- 只能填写"内容"列，其他部分（检查要求、状态列、表头结构）一律不得修改
- 格式克隆：打开.docx模板 → 只填空白单元格 → 100%保留模板原文
- 绝对禁止：添加颜色、加粗、项目符号、分隔线、调整段落缩进/行距/字体大小、增删空格或换行符
- ISO20000模板：CEPREI A-196-D-2024.08（45表，6列）
- ISO27001模板：CEPREI A-152-M-2024.08（17表，6列）

### Rule 3：内容列必须包含制度文件摘抄

- 格式："[制度文件名]：[3-5句简要摘抄]。查[记录证据]..."
- 制度文件名必须从客户记录中实际存在的文件里引用，不能编造

### Rule 4：通用话术模板

- ISO27001话术：方针策略类/程序文件类/风险评估类/访问控制类/资产管理类/事件管理类/内审管理评审类/合规类
- ISO20000话术：服务管理类/过程管理类/服务级别类/变更发布类/事件问题类
- 每次先匹配话术模板，再填入客户具体制度文件名和记录内容

### Rule 5：审核验证话术（查项目/查记录的固定表述）

- 文件查阅类："查《XXX管理程序》，[3-5句摘抄]"
- 记录核查类："查XXX项目XXX记录，[证据摘要]"
- 现场/访谈类："访谈[岗位/人员]，了解了[具体内容]"
- 抽查/验证类："抽查[具体对象]，[结果]"
- 内容列完整结构：①制度文件描述 → ②查记录/项目 → ③访谈/抽查 → ④结论性语句
- 常用结尾："保留了相关记录"/"验证无异常"/"符合要求"/"经审批后执行"

### Rule 6：合同及信息安全记录查核的"前-中-后"话术

- 合同查核：前（合同对方+签订时间）→ 中（合同内容）→ 后（信息安全要求条款+结论）
- 项目记录查核：前（项目名+甲方+合同编号+期限）→ 中（查什么记录+内容）→ 后（核查结果+保密条款）
- 信息安全记录查核：前（记录年份/版本）→ 中（具体发现）→ 后（符合性结论）
- 访谈/抽查/查看系统：各有固定的"前-中-后"结构
- ISO27001特有：保密协议/权限审查/备份恢复/漏洞补丁/防病毒查核话术

### Rule 7：项目范围覆盖与一致性核查

- ISO27001项目抽样：集成→查2个（完成+进行中）；开发→查2个；运维→查1个
- ISO20000项目抽样：查运维项目，确认运维类型、执行期、术语一致性
- ⚠️ **合同核查范围（极重要！）**：
  - "三条线对齐"（合同↔营业执照↔认证范围）仅限综合部分"抽查项目"条款和A.5.8
  - 其他过程条款只查保密相关字眼，不查范围一致性
  - ❌ 在4.3/A.5.10/A.5.31等条款加三条线对齐 → 错误

### Rule 8：内审和管理评审核查规则

- 时间线：管理评审必须在内审不符合项关闭后进行，时间不能重合
- 人员：内审可由管理者代表执行；管理评审必须由最高管理者主持
- 最高管理者确认：查管理手册授权书 → 授权人即最高管理者；查SoA → 批准人即最高管理者
- ISO27001风险评估6步全流程：①前期培训 → ②风险评估计划 → ③风险评估执行 → ④风险处置 → ⑤残余风险评估 → ⑥残余风险报告经最高管理者签字确认
- 常见问题：缺培训→不符合；缺残余风险评估→不符合；残余风险未经最高管理者签字→不符合

### Rule 9：自主学习与内容如实引用规则

- **写检查表前必须学习**：读取已有检查表（尤其内审/管理评审/风险评估条款），学习其表述方式、信息密度和话术风格
- **所有过程记录都必须参照客户实际记录来写，不限于内审/管理评审/风险评估**：
  - 每一个检查项的内容列，都必须从客户记录中提取真实信息
  - 制度文件名、记录编号、人员姓名、项目名称、合同金额等必须来自实际记录
  - ❌ 严禁编造、臆测、泛泛描述
  - 如果客户记录中缺少某项信息，用"查[记录名]"占位，不编造内容
- **内容具体化要求**（极重要，当前最需改进！）：
  - ❌ 错误示范："查《内部审核计划》，内审按计划执行。查《内审报告》，内审覆盖了所有部门和条款。"（太泛！无具体时间、人员、发现）
  - ✅ 正确示范："查《年度内部审核计划》(JL-06-01)，内审日期为2024年4月29-30日。查《内审实施计划》(JL-06-02)，内审覆盖了管理层、市场营销中心、服务交付中心、综合服务中心4个部门。查《内审报告》(JL-06-07)，内审发现1项不符合。查《内审不符合项报告》(JL-06-05)，不符合项已制定纠正措施。查《不符合整改验证记录》(JL-06-06)，2024年5月15日验证关闭。内审员具备内审员资质，且不审核本部门工作。验证无异常。"
- **五大信息维度**（每个检查项内容必须尽量覆盖）：
  1. **时间**：制度制定日期、审核日期、评审日期、记录时间、整改关闭日期等
  2. **人员**：审核员姓名、最高管理者姓名、管理者代表姓名、培训讲师等
  3. **地点**：办公地址、机房位置、项目实施地点等
  4. **发现**：不符合项数量及内容、管理评审发现的问题、风险数量及等级、改进项等
  5. **结果**：整改是否关闭、风险是否降级、目标是否达成、验证是否通过等
- **重点关注条款的具体化要求**：
  - 内审(9.2)：必须写明内审日期、覆盖部门数、不符合项数量、不符合内容概要、关闭日期、内审员资质
  - 管理评审(9.3)：必须写明评审日期、主持人、参会人员、输入内容概要、输出改进项、时间线与内审关系
  - 风险评估(8.2)：必须写明计划日期、培训日期、风险数量及等级、处置结果、残余风险、最高管理者签字确认
  - 合规性评价：必须写明评价日期、识别的法律法规数量、评价结论
  - 绩效评估(9.1)：必须写明目标具体数值、达成情况、监控频率
- **自主学习流程**（每次写检查表必须执行）：
  1. 读取客户记录文件（制度文件、记录文件、项目资料等）
  2. 从记录中提取所有具体信息（时间、人员、发现、结果等）
  3. 读取已有检查表样本，学习表述风格和信息密度
  4. 对照规则4~8的话术模板，将提取的真实信息填入内容列
  5. 所有信息必须有来源（客户记录或用户确认），不编造

### Rule 10：审核时间锚点与记录时效性校验规则

- **审核时间锚点**：
  - 写检查表前，必须先向用户确认审核计划的开始日期
  - 以审核计划日期为基准，往前倒推一年，形成**时间有效窗口**
  - 校验所有过程记录的时间是否在此一年窗口内：
    - ✅ 在一年窗口内 → 时间合格，可直接引用
    - ❌ 超出一年窗口 → **时间有问题**，必须暴露出来
  - 超出时间窗口的处理：
    1. 在**问题清单**中体现（明确标注哪项记录的时间超出审核有效窗口）
    2. 告知用户该记录时间不在有效期内，需要企业调整
    3. 或由用户提供一个合适的时间
    4. 如果既不调整也无合适时间，该记录的时间信息不可引用
  - **需要校验时效性的记录类型**（不限于内审/管理评审/风险评估，而是所有过程记录）：
    - 内审：内审计划日期、内审实施日期、不符合项整改关闭日期
    - 管理评审：管理评审计划日期、管理评审实施日期
    - 风险评估：评估计划日期、培训日期、评估实施日期、残余风险评估日期
    - 培训记录：培训实施日期
    - 合规性评价：评价实施日期
    - 管理体系文件：文件发布/修订日期
    - 项目记录：项目执行期间
    - 其他所有带时间的过程记录
- **预输出对话流程**（必须执行，不可跳过）：
  1. 读取并学习客户记录 → 提取所有可用的具体信息
  2. 读取并学习已有检查表的表述风格
  3. 向用户确认审核计划开始日期
  4. 校验所有过程记录的时间是否在有效窗口内
  5. **向用户展示以下信息并确认**：
     - 审核计划开始日期（用于时间锚点）
     - ⚠️ **时效性校验结果**：哪些记录在有效窗口内（✅）、哪些超出（❌）
     - 内审日期、管理评审日期、风险评估日期
     - 不符合项数量及内容
     - 管理评审发现的问题和改进项
     - 风险评估的识别数量、等级分布、处置结果
     - 其他关键过程信息的时间节点
  6. 用户确认或补充信息后，再输出最终的检查表文档
  7. 超出有效窗口的记录，在问题清单中体现
- **此规则的优先级高于直接输出**：即使已完成了所有内容填写，也必须先与用户确认再输出

### Rule 11：记录信息如实引用规则（极重要！真的去查记录信息！）

- **核心要求：你必须真的去查客户记录里的信息，不能只是仿写格式！**
  - 文件名：必须与客户记录中实际存在的文件名完全一致（如"CR-ITSMS-B-01服务目录管理程序"，不能写"相关管理程序"或另一个企业的文件名）
  - 记录编号：必须从实际记录中提取（如"QC-4.5.4.3-01"），不能自己编造
  - 人员姓名：必须从记录中确认（如内审员"华正凯"、总经理"于洪文"），不能用其他企业的姓名
  - 项目名称：必须与合同/项目资料中的名称一致（如"CPX24005-海关视频监控系统运维项目"）
  - 合同编号/金额/期限：必须来自实际合同
  - 日期：必须来自实际记录，不能用估算值
- **❌ 绝对禁止**：
  - 用其他企业的制度文件名、记录编号、人员姓名（张冠李戴！写出来就成了另一个企业的信息！）
  - 编造不存在的记录编号
  - 用泛化的文件名（如"相关管理程序"）替代具体的文件名
  - 不查记录就凭感觉写，导致信息完全无法匹配该企业
- **验证方法**：每个检查项内容列中引用的文件名/编号/人名/日期，都必须能在客户记录文件中找到对应
- **如果记录中缺少某项信息**：用"查[记录名]"表述，不编造具体内容

### Rule 12：问题发现与问题清单输出规则（写检查表不是主要目的，发现问题才是！）

- **核心认知：读记录发现问题 > 写检查表**
  - 写检查表不是最主要的目的，在读记录的过程中发现问题、提出问题才是核心价值
  - 每次读记录时，必须带着"发现问题"的意识去读，不能只是机械地填写检查表
  - 问题清单和整改建议的优先级高于检查表本身
- **读记录的过程中必须主动发现问题**：
  - 时间问题：记录时间超出审核有效窗口（规则10）、时间线不合理
  - 记录不全：缺少必要的记录（如缺内审不符合项报告、缺风险评估培训记录等）
  - 逻辑问题：时间线不合理（如管理评审早于内审不符合项关闭）
  - 人员问题：管理评审不是最高管理者主持、内审员审核本部门工作等
  - 内容问题：制度文件与实际执行不一致、记录内容不完整等
  - 合规问题：不符合标准要求的情况
- **每次输出检查表时必须同时输出**：
  1. **检查表.docx**（格式克隆，按规则1~10）
  2. **问题清单.docx**，包含：
     - 序号 | 问题描述 | 涉及条款/过程 | 问题类型（时效性/完整性/逻辑性/合规性） | 严重程度（高/中/低） | 整改建议
  3. **信息缺失清单**，包含：
     - 序号 | 缺失信息 | 涉及条款/过程 | 影响程度 | 建议补充方式
  4. **整改建议与改进建议**，包含：
     - 对企业的整改建议（必须整改项）
     - 对企业的改进建议（可以提升的方面）
     - 需要企业补充的资料/记录清单
- **问题清单必须基于实际记录中发现的问题**，不能凭空编造问题
- **检查表输出时要暴露**：
  - 哪些信息已确认、哪些信息未收集全、哪些信息有问题需要关注
  - 在检查表内容中标注"（待确认）"或"（日期待确认）"来暴露不确定信息

### Rule 14：检查内容时间具体化规则（极重要！每次检查项必须体现时间）

- **检查表中涉及时间的每一个记录，检查内容里必须有时间日期**：
  - 有确切日期的，必须写到年月日（如"2024年3月18日"）
  - 日期不好确定的，**至少写到月份**（如"2024年3月"）
  - ❌ 绝对禁止：只写"已执行""按计划完成"等无时间描述
- **适用范围**：每一个检查项的内容列中，所有提到"查XXX记录""查XXX报告"的地方，都必须有时间
- **时间来源**：必须从客户记录中实际提取，不能用估算值
- **时效性校验**：写完时间后，对照规则10的时间有效窗口，校验是否在窗口内
- **常见必须写时间的场景**：
  - 制度文件发布/修订：写发布日期或修订日期（至少到月份）
  - 内审：写内审实施日期（年月日）、不符合项关闭日期（年月日）
  - 管理评审：写评审实施日期（年月日）
  - 风险评估：写评估实施日期（年月日）
  - 培训：写培训实施日期（至少到月份）
  - 合规性评价：写评价实施日期（至少到月份）
  - 合同签订：写签订日期（至少到月份）
  - 项目执行：写项目服务期限起止时间
  - 监控/测量：写监控周期或报告日期（至少到月份）
  - 变更记录：写变更实施日期（至少到月份）
  - 事件/问题记录：写事件发生日期（至少到月份）
- **如果记录中找不到具体时间**：
  - 优先从文件名、文件属性、内容中推断
  - 实在无法确定的，写"查[记录名]（日期待确认）"并记入信息缺失清单
  - 不能因为"不确定"就跳过不写时间
